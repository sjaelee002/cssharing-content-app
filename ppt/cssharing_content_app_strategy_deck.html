Pipeline Overview

Input Design

사용자 입력: 쉬운 필드와 고급 옵션을 분리

일반 사용자가 전략적 원인이나 오해를 바로 작성하기 어려울 수 있기 때문에, 필수 입력은 최소화하고 나머지는 AI의 도움을 받아 Master Brief 생성할 수 있도록 설계

Primary Input

항상 보이는 입력값

콘텐츠 주제예: 성수기 쇼핑몰 CS대응
타겟 독자운영팀장, 쇼핑몰 대표, CS 담당자
주요 키워드줄바꿈 또는 콤마로 입력
글 유형개념정의형 / 비교형 / 문제해결형 등
산업군이커머스, 쇼핑몰, 렌탈 등
연결 솔루션체크박스로 다중 선택
Advanced Options

비워둬도 되는 고급 입력값

  • 독자 상황: AI가 주제·산업군·솔루션을 기반으로 추천
  • 반박할 오해: 표면적 해결책을 AI가 추론
  • 구조적 원인: 운영 구조 관점으로 보완
  • 콘텐츠 목표: 기본값은 검색 유입 + 상담 고려 유도
  • CTA 유형: 기본값은 운영구조 상담
  • 섹션 수: 4~6 범위만 허용. Default는 5개
avoid expressions는 사용자에게 묻지 않고 prompt의 고정 안전규칙으로 처리
Master Content Brief

Master Brief: 4개 채널로 뻗기 전의 공통 전략

Master Brief는 4개 채널로 분리 전 “무엇을 말할지”를 정리하는 채널 중립 레이어.

사용자는 카드형 editor에서 내용을 수정가능하고 JSON에 반영할 수 있음.

주요 필드

core_problem reader_context misconception structural_cause core_message solution_links cta_direction channel_strategy avoid_notes
  • 채널별 문체가 아니라 공통 메시지와 논리 구조를 정리
  • 각 채널별 outline generator가 참고할 전략 JSON 생성
  • 빈 고급 입력값은 AI가 추천/가정 수준으로 보완

왜 필요한가

  • 네이버 블로그, 매거진, 카드뉴스, 링크드인의 메시지 일관성 확보
  • 유저가 먼저 공통 전략을 검수한 뒤 채널별 생성 가능
  • 향후 reviewer가 채널별 톤과 공통 메시지 충돌 여부를 점검할 수 있음
현재 검증: Master Brief의 core_message를 수정한 뒤 블로그 outline 생성 시 해당 방향이 반영됨

Backend Architecture

Backend: FastAPI가 Python 서비스와 프롬프트를 연결

React는 화면과 입력을 담당하고, FastAPI는 API key 처리, prompt 조립, Claude API 호출, 결과 반환을 담당한다.

csssharing_content_app 폴더에서 uvicorn app:app --reload --host 0.0.0.0 --port 8000

Frontend Architecture

Frontend: React + Vite 기반 단계형 편집 UI

프론트엔드: 사용자가 AI 결과를 검토하고 수정하는 작업 공간

cssharing_content_app/frontend 폴더에서 npm run dev

Step 1. 사용자 입력

Primary fields + Advanced Options. 어려운 필드는 비워두면 Master Brief에서 추론.

Step 2. Master Brief

카드형 editor로 수정 가능. Advanced JSON은 Apply 버튼으로 반영.

Step 3. 채널별 Outline

현재 네이버 블로그 구현 완료. 다른 채널은 버튼 구현 예정.

Deployment Path

현재 localhost에서 웹/앱 형태로 확장하는 방향

현재는 로컬 개발 서버지만, 구조는 배포 가능한 웹앱 형태로 발전 목표로 함

1

Local MVP

localhost:5173 React UI, localhost:8000 FastAPI. 현재 구현 단계.

2

Team Demo

private GitHub checkpoint, 로컬 시연 또는 화면 녹화. API key는 사용자가 세션마다 입력.

3

Web Deployment

Frontend는 Vercel/Netlify, Backend는 Cloud Run/Render/Railway 등으로 분리 배포 가능.

4

App-like Packaging

필요 시 Electron/Tauri 검토. 단, Python backend 패키징 복잡도 때문에 후순위.

우선순위는 desktop 앱 패키징보다 웹 배포와 팀원이 쓰기 쉬운 UI 안정화다.
